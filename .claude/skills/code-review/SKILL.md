# Code Review Skill

코드를 Critical과 Minor 두 단계로 분류해 리뷰한다.
개발자의 설계 의도를 존중하되, 문제가 될 수 있는 부분을 명확하게 짚는다.

---

## 원칙

- 리뷰는 사실에 근거한다. 추측성 지적은 하지 않는다.
- 문제의 심각도를 명확히 구분한다.
- 변수 네이밍은 명확하게 할 것. 과도한 생략과 축약을 하지 않는다.
- 개선 방향을 제시할 때는 간결한 예시 코드를 함께 제공한다.
- 잘 작성된 부분은 언급하지 않아도 된다. 리뷰는 개선에 집중한다.

---

## 심각도 분류

### 🔴 Critical

반드시 수정해야 하는 문제. 배포 전에 해결해야 한다.

| 항목 | 설명 |
|---|---|
| 요구사항 불충족 | 기능이 요구사항을 만족하지 않음 |
| 버그 가능성 | 잘못된 로직, 조건 오류, null 처리 누락 등 |
| 부족한 예외 처리 | 예외가 전파되거나 무시되어 장애로 이어질 수 있는 경우 |
| 트랜잭션/동시성 오류 | 트랜잭션 범위 오류, 경쟁 조건, 낙관적/비관적 락 미적용 |
| 책임 분리 문제 | 기술 부채로 번질 가능성이 있는 레이어 간 책임 혼재 |

### 🟡 Minor

개선하면 좋은 문제. 당장 배포를 막을 수준은 아니지만 코드 품질에 영향을 준다.

| 항목 | 설명 |
|---|---|
| 테스트 가능성 | 의존성이 하드코딩되어 테스트하기 어려운 구조 |
| 불분명한 네이밍 | 의도를 알기 어려운 변수명, 메서드명, 클래스명 |
| 코드 스멜 | 불필요한 주석, 중복 코드, 과도하게 긴 메서드 등 |

---

## 출력 형식

```
## 코드 리뷰

### 🔴 Critical

**[파일명 또는 메서드명]**
- 문제: 구체적으로 무엇이 문제인지 설명
- 이유: 왜 문제인지 (버그 발생 조건, 장애 시나리오 등)
- 개선 방향:
  ```java
  // 개선 예시 코드
  ```

---

### 🟡 Minor

**[파일명 또는 메서드명]**
- 문제: 간결하게 설명
- 개선 방향: 한두 줄로 제안 (필요 시 예시 코드)
```

Critical이 없으면 해당 섹션을 생략한다. Minor도 마찬가지다.

---

## 리뷰 체크리스트

코드를 받으면 아래 순서로 검토한다.

**Critical 검토**
- [ ] 요구사항을 모두 구현했는가
- [ ] null / 빈 값 처리가 누락된 곳이 없는가
- [ ] 예외가 적절하게 처리되고 있는가 (무시되거나 의도치 않게 전파되지 않는가)
- [ ] 트랜잭션 범위가 올바른가 (너무 넓거나 누락되지 않았는가)
- [ ] 동시성 문제가 발생할 수 있는 구간이 없는가
- [ ] 레이어 간 책임이 적절하게 분리되어 있는가 (Controller가 비즈니스 로직을 갖지 않는가 등)

**Minor 검토**
- [ ] 외부 의존성을 주입받는 구조인가 (테스트 가능성)
- [ ] 변수명, 메서드명, 클래스명이 의도를 명확하게 드러내는가
- [ ] 불필요한 주석이 있는가 (코드로 이미 명확한 내용을 주석으로 설명하는 경우)
- [ ] 중복 코드나 과도하게 긴 메서드가 없는가
