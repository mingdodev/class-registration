# Documentation Sync Skill

코드 변경분을 분석해 기존 문서와의 불일치를 탐지하고, 문서를 최신 상태로 수정한다.

---

## 원칙

- 코드가 기준이다. 문서와 코드가 충돌할 경우 코드를 따른다.
- 개발자의 설계 의도를 왜곡하지 않는다. 불명확한 변경 의도는 반드시 질문한다.
- 문서에 없는 내용을 임의로 추가하지 않는다. 새로 추가된 항목은 개발자에게 확인 후 반영한다.

---

## 탐지 대상

아래 변경 유형을 우선적으로 확인한다.

| 변경 유형 | 확인 항목 |
|---|---|
| API 변경 | 엔드포인트 추가/수정/삭제, HTTP 메서드 변경, 요청/응답 필드 변경, 상태코드 변경 |
| DTO 변경 | 필드 추가/삭제/타입 변경, validation 어노테이션 변경 |
| Domain Model 변경 | 엔티티 필드 추가/삭제, 관계 변경, 제약 조건 변경 |
| 환경 변수 변경 | `application.yml` / `.env` 에 변수 추가/삭제/키 이름 변경 |
| 실행 가이드 변경 | 실행 명령어 변경, 프로파일 변경, 사전 조건 변경 |

---

## 작업 흐름

### 1. 변경 범위 파악

개발자가 변경된 코드 또는 diff를 제공하면, 아래를 확인한다.

- 어떤 파일이 변경되었는가
- 각 변경이 어느 문서 유형에 영향을 주는가

### 2. 불일치 탐지

기존 문서와 코드를 대조해 불일치 항목을 목록으로 정리한다.

**출력 형식 예시**

```
[불일치 탐지 결과]

1. API 명세 (API.md)
   - POST /api/v1/users 응답 필드에 `nickname` 추가됨 → 문서에 없음
   - 에러코드 400의 설명이 코드와 다름

2. 실행 방법 (README.md)
   - 환경 변수 `MAIL_HOST` 추가됨 → 문서에 없음
```

### 3. 수정 전 확인

불명확한 변경 의도가 있는 경우, 수정 전에 개발자에게 질문한다.

예:
- "이 필드는 항상 응답에 포함되나요, 조건부인가요?"
- "이 환경 변수는 필수인가요, 선택인가요?"

### 4. 문서 수정

확인된 불일치를 반영해 문서를 수정한다.
수정 후 변경 요약을 간단히 제공한다.

**수정 요약 예시**

```
[수정 완료]

API.md
- POST /api/v1/users 응답 필드에 `nickname` (String) 추가
- 에러코드 400 설명 수정: "유효성 검사 실패" → "요청 형식 오류"

README.md
- 환경 변수 `MAIL_HOST` 추가 (SMTP 호스트 주소)
```

---

## 주의사항

- 문서에 의도적으로 생략된 내용(내부 구현 세부사항 등)은 임의로 추가하지 않는다.
- 여러 문서에 걸쳐 영향을 주는 변경이라면 모든 관련 문서를 함께 확인한다.
- 수정일을 기록하는 문서라면, 필요 시 수정 일자를 최신으로 업데이트한다.
