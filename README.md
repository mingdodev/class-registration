# 수강 신청 시스템

강사가 강의를 개설하고 수강생이 원하는 강의에 수강 신청하는 수강 신청 시스템 구현 과제입니다.  
정원 초과 시 대기열 등록, 스케줄러 기반 자동 상태 전이, 동시성 제어를 포함합니다.

<br>

---

## 기술 스택

| 항목 | 내용                                             |
|---|------------------------------------------------|
| 언어 | Java 21                                        |
| 프레임워크 | Spring Boot 4.0.6                              |
| 빌드 | Gradle                            |
| 데이터베이스 | MySQL 8.0 (stage) / H2 인메모리 (local) |
| 테스트 | JUnit 5                                        |

<br>

## 실행 방법

### 개발 환경 (local)

H2 인메모리 데이터베이스를 사용합니다. 별도의 설정 없이 애플리케이션을 실행할 수 있습니다.

```bash
./gradlew bootRun
```
애플리케이션은 기본적으로 `http://localhost:8080`에서 실행됩니다.  
H2 콘솔은 아래 주소에서 확인할 수 있습니다.
- `http://localhost:8080/h2-console`
- JDBC URL: `jdbc:h2:mem:class-registration`

### 스테이징 환경 (stage)

Docker Compose를 사용해 MySQL과 애플리케이션을 함께 실행합니다.

```bash
docker compose up --build
```

> 환경 변수는 편의를 위해 `docker-compose.yml`에 직접 정의되어 있습니다.

## 요구사항 해석 및 가정

도메인 규칙의 전체 내용은 [docs/DOMAIN.md](docs/DOMAIN.md)에 정리되어 있습니다.  
요구사항 해석 과정에서 추가적인 결정이 필요했던 주요 사항은 다음과 같습니다.

- **수강 기간 무제한**
    - `startDate`, `endDate`를 둘 다 설정하지 않으면 수강 기간에 제한이 없는 것으로 해석
    - 두 날짜는 항상 함께 설정하거나 둘 다 생략해야 함

- **수강 취소(CANCELLED) 후 재신청 허용**
    - 취소된 수강 신청은 이력으로 남기되, 동일 강의에 재신청 가능
    - 유니크 제약은 CANCELLED를 제외한 수강 신청 컬럼에만 적용

- **결제 대기(PENDING) 상태의 취소 기간 제한 없음**
    - 결제 대기 중인 수강 신청은 수강 시작일과 무관하게 언제든 취소 가능
    - 취소 기간 제한은 결제 확정(CONFIRMED) 상태에만 적용

- **수강 확정(CONFIRMED) 상태의 취소 가능 기간**
    - 강의 시작일 3일 전까지 취소 가능
    - 강사의 강의 준비 기간 고려

- **강의별 수강생 목록 조회 범위**
    - 취소를 제외한 결제 대기, 수강 확정 상태의 수강 신청을 포함

- **대기열 운영 조건**
    - 신청 가능 인원이 0이 되어 마감(CLOSED)된 강의에 한해 대기열 등록을 허용
    - 수강 종료일이 지난 강의는 대기열 등록 불가

- **수강 신청 가능 인원 복구 시 대기열 우선 처리**
    - 잔여 인원이 발생하더라도 즉시 모집 중(OPEN)으로 전환하지 않고, 대기열을 먼저 처리
    - 대기열이 모두 소진된 이후에도 잔여 인원이 남아 있을 경우에만 모집 중(OPEN) 상태로 전환

- **페이지네이션 적용 범위**
  - 데이터 규모가 커질 수 있는 API에만 커서 기반 페이지네이션 적용
      - 전체 강의 목록
      - 내 수강 신청 목록

## 설계 결정과 이유

핵심 설계 결정은 아래와 같으며, 구체적인 전체 의사결정 기록은 하단의 ADR 문서에서 확인할 수 있습니다.

### 1. 수강 신청 동시성 제어

수강 신청은 다수의 수강생이 동시에 요청하는 상황을 고려하여, **DB 수준의 원자적 업데이트**를 사용해 처리했습니다.

```sql
UPDATE klasses
SET remaining_capacity = remaining_capacity - 1
WHERE id = ? AND remaining_capacity > 0
```

원자적 업데이트는 조건 검사와 차감이 한 번에 수행되므로, 동시 요청 상황에서도 정원 초과 없이 정합성이 보장됩니다. 또한 락 점유 시간이 짧아 성능 측면에서도 유리합니다.

낙관적 락과 비관적 락 모두 대안으로 적용 가능하지만,
트래픽이 증가하거나 동시 요청이 많아질 경우 재시도 비용 또는 락 대기로 인해 성능 저하가 발생할 수 있다고 판단했습니다.

원자적 업데이트는 일부 비즈니스 로직(정원 검사 및 차감)이 애플리케이션 코드가 아닌 DB 쿼리에 포함되는 트레이드오프가 있지만,
책임의 분리보다는 성능과 안정성을 우선하여 선택했습니다.

### 2. 스케줄러 기반 상태 전이와 요청 시점 보완

수강 신청 후 24시간 내 결제하지 않으면 자동 취소(PENDING → CANCELLED)되고,  
수강 종료일이 지나면 강의는 모집 마감(OPEN → CLOSED) 상태로 전이됩니다.  
이러한 상태 전이는 **애플리케이션 내 스케줄러**로 구현했습니다.

또한 스케줄러로 주기적으로 처리하되, **요청 시점에도 만료 여부를 재확인**하는 이중 방어 전략을 사용했습니다.
스케줄러만으로는 실행 주기 사이에 들어오는 요청을 막을 수 없기 때문입니다.
예를 들어 결제 기한이 지난 수강 신청이 스케줄러 실행 전에 결제 확정을 요청하거나, 수강 종료일이 지난 강의에 신청이 들어오는 상황이 발생할 수 있습니다. 요청 시점의 재확인이 이를 보완합니다.

DB 내장 스케줄러(MySQL Event Scheduler)도 대안이었지만, 상태 전이라는 핵심 비즈니스 로직이 DB에 흩어지고 개발 및 테스트 환경인 H2에서 지원하지 않아 제외했습니다.

### 3. 대기열 처리

잔여 인원이 0에서 1 이상으로 복구될 때(수강 취소 또는 정원 증가),  
대기열에 있는 수강생에게 순서대로 수강 신청(PENDING)을 생성하는 처리가 필요합니다.

이를 위해, 잔여 인원이 복구되는 시점에 **대기열 처리 이벤트를 발행**하고, **소비자가 이를 순차적으로 처리**하는 방식으로 설계했습니다.

이벤트는 인메모리 `ConcurrentLinkedQueue`로 관리하며, 이벤트를 소비하는 로직은 Java 21 가상 스레드를 사용해 요청 스레드를 블로킹하지 않고 백그라운드에서 처리합니다.

강의별로 이벤트 큐를 분리하여, 특정 강의의 처리 실패가 다른 강의에 영향을 주지 않도록 했습니다.  
이벤트는 처리에 성공한 경우에만 큐에서 제거(`peek → poll`)되므로, 실패 시 이벤트가 유실되지 않습니다.

또한 3회 연속 실패가 발생하면 서킷 브레이커가 해당 강의의 이벤트 처리를 일시 중단하고,
5분 주기 스케줄러를 통해 재시도를 수행합니다.

Kafka, RabbitMQ 같은 외부 메시지 큐는 영속성과 장애 복구 측면에서 더 안정적이지만,
현재 규모에서 외부 인프라를 추가하는 것은 과한 구현이라 판단했습니다.
트레이드오프로 서버 재시작 시 처리 중이던 이벤트가 유실될 수 있습니다.

## 미구현 / 제약사항

### 인증/인가 제약

- 사용자 인증 정보를 토큰/세션으로 유지하지 않고, 사용자 ID(`Creator-Id`, `Klassmate-Id`)를 요청 헤더로 전달하는 방식으로 구현했습니다.
- 강사와 수강생의 데이터를 분리했기 때문에 역할 기반 인가는 적용하지 않았으며,
  자신이 소유한 리소스에 대한 인가만 구현했습니다.

### 결제 대역을 사용

- 외부 결제 API 대신, 가짜 결제 클라이언트를 사용하여 `true` 반환 시 수강 신청 상태를 PENDING에서 CONFIRMED로 변경하도록 구현했습니다.
- 결제 정보를 별도로 저장하지 않으며, 이에 따라 다음 기능은 미구현 상태입니다.

    - 강의 폐강 및 강의 폐강 시 자동 결제 취소
    - 강의 오픈 이후 가격 변경 (기존 결제 금액 추적 불가)

### 수강 기간의 도메인 규칙 제약

- 모집 기간을 별도로 두지 않고, 수강 기간과 정원 상태에 따라 모집 상태가 결정되도록 설계했습니다.  
  요구사항 상 모집 기간을 분리하지 않아도 상태 전이로 충분히 표현 가능하다고 판단하여 단순화했습니다.
- 모집 상태의 예측 가능성을 유지하기 위해, 강의를 공개(OPEN, CLOSED)한 이후에는 수강 기간 변경을 제한했습니다.
- 수강생에게 유리한 변경(종료일 연장, 무제한 전환)이 구현되지 않았다는 한계가 있습니다.

### 대기열 처리 안정성

현재 구현은 `peek → poll` 방식으로, 처리 성공 전까지 이벤트를 큐에 유지하며 실패 시 1회 재시도,
연속 3회 실패 시 서킷 브레이커를 통해 처리를 일시 중단하고 5분 주기 스케줄러를 통해 재시도를 수행합니다.

그러나 다음과 같은 한계가 존재합니다.

- **이벤트 유실 시 대기열 처리 지연**

  인메모리 큐를 사용하므로 서버 재시작 또는 `publish()` 실패 시 대기열 처리 이벤트가 유실될 수 있습니다.  
  이 경우 잔여 인원은 복구되었지만 대기열 처리가 실행되지 않아, 대기열에 있는 수강생에게 PENDING 신청이 생성되지 않습니다.  
  또한 대기열 처리 과정에서 수행되어야 할 강의 상태 보정도 실행되지 않아, 잔여 인원이 존재함에도 CLOSED 상태로 유지되는 불일치가 발생할 수 있습니다.

  이를 보완하기 위해 잔여 인원이 존재하는 CLOSED 강의를 주기적으로 탐지하여 이벤트를 재발행하는 보정 스케줄러를 둘 수 있으나, 현재는 구현하지 않았습니다.

- **트랜잭션 롤백 시 불필요한 서킷 차단 가능성**

  이벤트를 트랜잭션 커밋 이전에 발행하기 때문에, 롤백이 발생해도 이벤트는 큐에 남아 있습니다.  
  이로 인해 소비자가 실제로는 처리할 수 없는 이벤트를 반복적으로 시도하게 되고,  
  실패가 누적되면 서킷 브레이커가 불필요하게 동작할 수 있습니다.  
  `@TransactionalEventListener(phase = AFTER_COMMIT)`을 적용하면 개선 가능하지만 현재는 미구현 상태입니다.  
  다만, 현재 구조에서도 비즈니스 로직의 정합성에는 문제가 발생하지 않습니다.

- **영구 오류 발생 시 즉시 인지 불가**

  재시도 실패 시 로그만 남기므로 실시간 알림 체계가 없습니다.  
  운영 환경에서는 알림 발송 등 즉각적인 대응 체계가 필요합니다.
