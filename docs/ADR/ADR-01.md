## 1. Creator / Klassmate 엔티티 분리

**상태**: 결정됨

### 컨텍스트
강사(Creator)와 수강생(Klassmate)을 단일 `User` 테이블로 관리할지, 별도 엔티티로 분리할지 결정이 필요했다.

### 결정
별도 엔티티로 분리한다.

### 이유
1. LiveKlass는 나만의 온라인 강의 사이트를 만들 수 있는 플랫폼으로, 서비스의 특성을 고려하면 강사와 수강생의 User Flow를 완전히 분리하는 게 자연스럽다. 
2. 강사는 정산 계좌 등 수강생과 속성이 달라야 하는 부분이 있고, 그 차이는 더 커질 가능성이 있다 판단했다. 단일 테이블로 관리하면 한쪽에만 필요한 컬럼이 계속 생겨 테이블이 지저분해진다.

### 고려한 대안
| 대안 | 제외 이유 |
|---|---|
| `User` 단일 테이블 + `role` 컬럼 | 강사/수강생 속성 차이가 커서 불필요한 컬럼이 증가함 |

### 결과
도메인 모델뿐만 아니라 엔티티도 Creator / Klassmate로 분리한다.

## 2. 신청 가능 인원(remainingCapacity) 저장 방식

**상태**: 결정됨

### 컨텍스트
정원 초과 여부의 판단 기준이 될 컬럼으로 **현재 신청 인원**을 집계할지, **신청 가능 인원**을 집계할지 결정이 필요했다.

### 결정
신청 가능 인원(`remainingCapacity`)을 Klass 테이블에 직접 저장한다.

### 이유
수강 신청 시 `remainingCapacity > 0` 확인과 차감을 한 번에 처리할 수 있어 별도 집계 쿼리가 필요 없다.
필수 기능인 현재 신청 인원을 직접 저장하면 별도 연산 없이 바로 노출할 수 있지만, 반대로 수강 신청 시점마다 집계 쿼리가 발생한다.

### 고려한 대안
| 대안 | 제외 이유 |
|---|---|
| 현재 신청 인원 집계 쿼리 | 신청 시마다 COUNT 쿼리가 필요하고, 정원 초과 판단 로직이 복잡해짐 |

### 결과
신청 가능 인원을 저장했으니, 현재 신청 인원은 `maxCapacity - remainingCapacity`로 조회 시점에 연산한다.

## 3. 취소 사유(cancellationReason) 별도 필드

**상태**: 결정됨

### 컨텍스트
취소 상태(CANCELLED)만으로는 취소 경위를 알 수 없어 사용자 경험과 추적에 문제가 생길 수 있다.

### 결정
`Enrollment`에 `cancellationReason` 필드를 추가한다.

### 이유
결제 기한 초과 자동 취소, 직접 취소, 폐강으로 인한 취소는 사용자에게 다르게 표시되어야 한다. 필드 하나로 해결되며 나중에 추적 로직을 별도로 추가할 필요가 없다.

### 고려한 대안
| 대안 | 제외 이유 |
|---|---|
| CANCELLED 상태 세분화 | 상태가 늘어날수록 전이 규칙이 복잡해짐 |

### 결과
취소 사유 값: `PAYMENT_TIMEOUT`(결제 기한 지남), `USER_REQUESTED`(직접 취소), `KLASS_CLOSED`(폐강)

## 4. 대기열 설계 및 구현 범위 결정

**상태**: 결정됨

### 컨텍스트
정원이 마감된 강의에 대한 대기열 기능을 어떻게 설계하고, 처리 실패 시 어떻게 대응할지 결정이 필요했다.

### 결정

**트리거**: CLOSED → OPEN 이벤트가 아닌 수강 신청 취소 이벤트가 직접 대기열 처리를 트리거한다.

**트랜잭션 분리**: 취소 처리와 대기열 처리는 독립적인 트랜잭션으로 수행한다. 대기열 처리 실패가 취소에 영향을 주어서는 안 된다. 취소 트랜잭션 커밋 후 이벤트를 발행한다.

**대기열 처리**: 대기열 1순위 수강생의 수강 신청을 PENDING 상태로 생성한다.

**백그라운드 처리**: 대기열 처리는 요청 스레드를 블로킹하지 않도록 백그라운드에서 수행한다. Java 21 가상 스레드를 활용한다.

**실패 처리**: 대기열 처리 실패 시 1회 재시도한다. 재시도 후에도 실패하면 수강생, 강의 정보를 포함한 에러 로그를 남긴다.

### 이유
CLOSED → OPEN 이벤트를 트리거로 사용하면 그 순간 일반 수강 신청도 열려 대기열 순서를 앞지르는 새치기가 발생할 수 있다. 취소 이벤트를 직접 트리거로 사용하면 재고를 일반에 공개하기 전에 대기열을 먼저 처리할 수 있다.

### 고려한 대안
| 대안 | 제외 이유                                    |
|---|------------------------------------------|
| CLOSED → OPEN 이벤트 트리거 | 대기열 순서 새치기 발생 가능                         |
| 카프카 기반 이벤트 큐 + DLQ | 실운영이 아닌 과제 수준에서는 과한 구현이라고 판단, 미구현 범위에 포함 |
| 플랫폼 스레드 | 가상 스레드 대비 무겁고, Java 21 환경에서 이점이 없음       |

### 결과
재시도 1회 후에도 실패하면 신청 기회가 소실되고 재고 복원이 되지 않는 문제가 있다.

**미구현 범위**: 이벤트가 소실되지 않고 오류 복구가 가능한 구조가 필요하다.
취소 이벤트를 발행하고 이를 소비할 때 대기열 1순위 수강생의 수강 신청을 PENDING 상태로 생성한다.
소비 중 재시도로도 해결되지 않는 오류가 발생하면 DLQ로 이동시켜 별도로 모니터링하고 재처리해야 한다.
또한, 재시도 실패 시 개발자가 즉시 인지할 수 있는 구조가 필요하다. (알림 발송 등)
과제에서는 단순 재시도 1회 + 로깅으로 구현하였다.
