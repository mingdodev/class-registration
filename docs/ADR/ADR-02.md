## 1. 수강 신청 및 대기열 중복 방지 전략

**상태**: 결정됨

### 컨텍스트
수강생이 동일한 강의에 수강 신청 또는 대기열을 중복으로 요청하는 경우를 방지해야 한다.
UI 수준의 제어만으로는 부족하고, 데이터 정합성을 보장하는 수단이 필요했다.

### 결정
UI와 DB 두 단계로 중복을 방지한다.

**수강 신청**
- 신청 여부 확인 엔드포인트를 제공해 UI에서 1차로 재신청을 막는다.
- DB에 조건부 유니크 인덱스를 적용한다. CANCELLED 상태를 제외한 `(klassmate_id, klass_id)` 조합에 유니크 제약을 건다. 취소된 수강 신청은 재신청이 가능해야 하기 때문이다.

**대기열**
- 등록 여부 확인 엔드포인트를 제공해 UI에서 1차로 중복 등록을 막는다.
- DB에 `(klassmate_id, klass_id)` 유니크 제약을 건다.

### 이유
UI 제어만으로는 동시 요청이나 비정상적인 접근을 막을 수 없다. DB 수준의 제약이 그러한 예측 불가한 사태를 막아준다.

수강 신청에 단순 유니크 인덱스를 적용하면 취소 후 재신청이 불가능해진다. CANCELLED 상태를 제외한 조건부 유니크 인덱스를 사용해 재신청을 허용하면서도 중복을 방지한다.

### 고려한 대안
| 대안 | 제외 이유                                                                                              |
|---|----------------------------------------------------------------------------------------------------|
| 서비스 레이어에서 중복 체크 쿼리 | SELECT 후 INSERT 사이에 다른 트랜잭션이 끼어드는 레이스 컨디션 발생 가능. 서비스 레이어에서 트랜잭션 처리를 할 수도 있지만, DB 수준에 비해 성능이 좋지 않음. |

### 결과
INSERT 시 유니크 제약 위반 예외를 잡아 409로 응답한다.
