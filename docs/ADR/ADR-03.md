## 1. 도메인 모델을 영속 객체로 사용

**상태**: 결정됨

### 컨텍스트
도메인 모델(비즈니스 로직을 담은 객체)과 JPA 엔티티(영속성 객체)를 분리할지, 통합해서 사용할지 결정이 필요했다.

### 결정
도메인 모델과 JPA 엔티티를 분리하지 않고, 도메인 모델을 영속 객체로 그대로 사용한다.

### 이유
분리 시 두 계층 간 변환 코드가 필요해 개발 속도가 느려지고 코드량이 늘어난다. 현재 프로젝트 규모에서 분리로 얻는 이점(영속성 기술 교체 용이성, 도메인 모델의 순수성)보다 비용이 크다고 판단했다.

### 고려한 대안
| 대안 | 제외 이유 |
|---|---|
| 도메인 모델 / JPA 엔티티 분리 | 변환 레이어(Mapper) 추가 비용이 현재 규모에서 불필요 |

### 결과
JPA 어노테이션이 도메인 모델에 직접 붙는다. 도메인 모델이 JPA에 의존하게 되어 기술 교체 시 영향을 받을 수 있다.
그러나 JPA는 표준 기술이며, 기술을 교체하는 경우도 거의 없기 때문에 트레이드오프는 거의 없다고 판단했다.
어노테이션이 직접 붙는 문제 역시 `META-INF/orm.xml`을 통해 JPA 설정을 XML로 따로 관리하는 방식으로 개선할 수 있으나, 따로 구현하지는 않았다.

## 2. 프로파일 분리 및 DB 초기화 전략

**상태**: 결정됨

### 컨텍스트
환경(로컬 개발, 테스트, 운영 검증)마다 DB와 초기화 방식이 달라야 한다. 각 환경에 맞는 설정과 데이터 초기화 전략이 필요했다.

### 결정

**프로파일 구성**

| 프로파일 | 환경 | DB |
|---|---|---|
| `local` | 로컬 개발 | H2 인메모리 |
| `stage` | 운영 검증 | MySQL |

**환경별 초기화 전략**

| 환경 | 스키마 초기화                                | 데이터 초기화 |
|---|----------------------------------------|---|
| local | `ddl-auto=create`                      | `data.sql` 초기 데이터 제공 |
| test | `ddl-auto=create`                  | 테스트 코드에서 직접 생성/삭제 (Fixture 패턴 고려) |
| stage | `schema.sql` 직접 관리 + `ddl-auto=validate` | 없음 |

### 이유
로컬과 테스트 환경에서는 H2와 ddl-auto로 빠른 개발 및 채점이 가능하다. 운영 검증 환경(stage)은 실제 MySQL 스키마를 명시적으로 관리하고, `validate`로 엔티티-스키마 불일치를 사전에 감지한다.

테스트 환경에서 `data.sql`을 사용하지 않는 이유는, 테스트 결과를 외부에 의존하지 않기 위해서이다. 테스트 코드에서 직접 데이터를 제어해, 어떠한 환경에서도 성공하는 테스트를 작성한다.

### 고려한 대안
| 대안 | 제외 이유                                 |
|---|---------------------------------------|
| 모든 환경에서 MySQL 사용 | 로컬 개발 시 DB 설치/설정 비용 발생, 빠른 개발과 검증에 불리 |

### 결과
stage 환경에서 `schema.sql`을 별도 관리해야 하므로, 스키마 변경 시 `schema.sql`과 JPA 엔티티를 함께 수정해야 한다.
