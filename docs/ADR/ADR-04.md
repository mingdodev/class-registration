## 1. 수강 신청 동시성 제어

**상태**: 결정됨

### 컨텍스트
동시에 여러 수강생이 마지막 자리에 신청할 경우 정원을 초과해서는 안 된다. 수강 정원을 확인하고, 업데이트하는 작업 사이 다른 작업이 끼어들면 안 된다.

### 결정
DB 수준의 원자적 업데이트를 사용한다.

```sql
UPDATE klasses
SET remaining_capacity = remaining_capacity - 1
WHERE id = ? AND remaining_capacity > 0
```

업데이트된 행 개수를 애플리케이션이 확인해 0이면 신청 거부, 1이면 성공으로 처리한다.

### 이유
낙관적 락은 경합이 자주 일어나지 않을 때 적합할 것인데, 강의 수강 신청의 경우 인기가 많은 강의는 오픈하자마자 많은 동시 요청이 들어올 것이기 때문에 맞지 않는다 생각했다.
또한, 재시도 로직을 따로 구현해야 하는 데다가 경쟁이 심할 수록 고려해야 할 요소가 많아진다.
비관적 락의 경우 직관적으로 구현할 수 있다는 장점이 있고, 비즈니스 로직을 코드에 그대로 녹이기도 구조적으로 적합하다.
그러나 이는 락 점유 시간이 다른 방식에 비해 비교적 길다. 지금은 애플리케이션에서 락을 잡은 사이 별다른 작업을 하지 않을 예정이므로 크게 상관 없을 수 있지만,
만약 트래픽이 많아지고 경쟁이 심해진다면 요청이 커넥션 풀을 점유한 채 대기하게 되어 오류로 이어질 수 있다.
원자적 업데이트는 동시성 제어를 DB 수준에서 처리해, 점유 시간이 매우 짧다.
또한, remaining_capacity 차감은 조건이 단순하기 때문에 원자적 업데이트를 수행하기 적합하다.

### 고려한 대안
| 대안 | 제외 이유                                                  |
|---|--------------------------------------------------------|
| 낙관적 락 (버전 컬럼) | DB 락이 없지만, 충돌 시 재시도 로직을 직접 구현해야 하고, 경합이 심해진다면 복잡도가 높아짐 |
| 비관적 락 (SELECT FOR UPDATE) | SELECT 시점부터 트랜잭션 종료까지 락을 비교적 길게 점유                     |

### 결과
도메인 로직이 DB 업데이트 방식에 의존하게 된다는 트레이드 오프가 있다. 그러나, 성능을 우선으로 고려하여 설계했다.

## 2. PENDING 자동 취소

**상태**: 결정됨

### 컨텍스트
수강 신청 후 24시간 이내 결제하지 않으면 자동으로 취소 처리해야 한다.

### 결정
두 단계로 처리한다.

- **스케줄러**: 1시간 주기로 PENDING 상태이고 신청 시각으로부터 24시간이 지난 행을 조회해 CANCELLED로 일괄 업데이트한다.
- **결제 요청 시점**: 결제 처리 전 만료 여부(신청 시각 + 24H <= 현재)를 확인해 만료된 건은 거부한다. 

### 이유

스케줄러만으로는 실행 주기 사이에 만료된 건에 대한 결제 요청을 막을 수 없다. 혹은 비정상적인 요청이 있을 수 있으므로,
결제 시점의 만료 확인으로 이를 보완한다.
1시간 주기는 초기값이며, 애플리케이션 부하에 따라 조정할 수 있다.

## 3. 강의 상태 자동 전환 (OPEN → CLOSED)

**상태**: 결정됨

### 컨텍스트
수강 종료일이 지난 모집 중(OPEN) 강의는 자동으로 모집 마감(CLOSED) 상태로 전환되어야 한다.

### 결정
두 단계로 처리한다.

- **스케줄러**: 24시간 주기로 OPEN 상태이고 수강 종료일이 지난 강의를 조회해 CLOSED로 일괄 업데이트한다.
- **수강 신청 요청 시점**: 수강 신청 전 강의의 수강 종료일이 지났는지 확인해 만료된 강의는 신청을 거부한다.

### 이유
스케줄러만으로는 실행 주기 사이에 종료일이 지난 강의에 대한 수강 신청을 막을 수 없다. 수강 신청 시점의 종료일 확인으로 이를 보완한다. PENDING 자동 취소와 동일한 이중 방어 전략이다.

> 2, 3 두 방식 모두 **애플리케이션 수준의 스케쥴러**를 채택한다.  
> 핵심 이유는, DB 내 스케줄링 방식은 비즈니스 로직이 DB에 과하게 의존하게 된다는 문제가 있기 때문이다.
> MySQL과 달리 H2에는 내장된 이벤트 스케줄러가 존재하지 않는다는 제약도 있다.

## 4. 대기열 이벤트 처리

**상태**: 결정됨

### 컨텍스트
잔여 인원이 0에서 1 이상으로 바뀌는 시점에 대기열을 처리해야 한다.  
처리 실패 시에도 순서가 보장되어야 하며, 수강생의 신청 기회가 사라져서는 안 된다.  
외부 메시지 큐 없이 구현한다.

### 결정

**이벤트 발생 조건**
- **CLOSED 상태에서** 수강 신청 취소 시 이벤트 발행
- **CLOSED 상태에서** 수강 정원 증가 시 증가한 인원 수만큼 이벤트 반복 발행 (1자리 복구 → 1회 발행)
- OPEN 클래스에서는 이벤트를 발행하지 않는다. 대기열은 CLOSED 상태에서만 존재하므로 처리가 불필요하다.

**처리 순서**
1. 잔여 인원 즉시 반영 — 취소 시 `remaining_capacity + 1`, 정원 증가 시 증가분만큼 즉시 반영
2. 이벤트 발행 (큐에 추가)
3. 소비자 스레드가 큐를 소비하며 대기열 수강생에게 PENDING 신청 자동 생성
4. 대기열 소진 후 잔여 인원이 남아있으면 스케줄러가 OPEN으로 전환

취소 처리와 대기열 처리는 독립적으로 수행된다. 이벤트 발행 실패가 취소에 영향을 주어서는 안 된다.

**큐 구조**
강의별로 독립된 인메모리 `ConcurrentLinkedQueue`를 사용한다. `ConcurrentHashMap<klassId, Queue>`로 관리하며, 첫 이벤트 도착 시 `computeIfAbsent`로 해당 강의의 큐를 생성한다. 맵 자체는 앱 시작 시 Spring 빈으로 등록된다.

강의별로 큐를 분리함으로써 한 강의의 처리 실패가 다른 강의에 영향을 주지 않는다.

**소비자 스레드**
강의별로 `AtomicBoolean`으로 실행 여부를 관리해 소비자 스레드가 강의당 하나만 생성되도록 보장한다.
이벤트 발생 시 해당 강의의 소비자가 실행 중이 아니면 가상 스레드를 생성해 큐를 비울 때까지 루프한다.
큐가 비면 스레드가 종료되고 다음 이벤트 발생 시 재생성된다.

**소비 방식: peek → 처리 성공 → poll**
이벤트를 큐에서 제거하지 않은 상태로 처리를 시도하고, 성공한 경우에만 제거한다. 실패 시 이벤트는 큐 헤드에 그대로 유지된다.

**장애 처리: 서킷 브레이커 패턴 적용 (강의별 독립)**

| 상태 | 동작 |
|---|---|
| CLOSED | 이벤트 드리븐으로 정상 소비 |
| OPEN | 소비 중단. 이벤트는 해당 강의 큐에 누적 |
| HALF-OPEN | 스케줄러가 주기적으로 큐 헤드 처리 시도. 성공 시 CLOSED 복귀, 실패 시 OPEN 유지 |

실패 임계값 초과 시 OPEN 전환. 일시적 오류는 HALF-OPEN에서 자동 복구되고, 영구 오류는 개발자가 수정 후 다음 스케줄러 실행 시 자동 복구된다.

### 고려한 대안
| 대안                         | 제외 이유                                                                                                                    |
|----------------------------|--------------------------------------------------------------------------------------------------------------------------|
| DLQ                        | 실패한 이벤트를 블로킹 없이 나중에 재처리할 수 있다는 이점이 있음. <br/>그러나, 규모가 작고 외부 모니터링이 없는 현재 구조에서 DLQ를 따로 두는 것은 이점보다는 복잡도가 증가한다는 단점이 더 크다고 판단. |
| 메시지큐(Kafka/RabbitMQ) + DLQ | 영속성을 보장하면서 장애 복구(DLQ, 재처리 등)를 편리하게 구현할 수 있지만, 현재 규모/환경에서 노드를 늘리는 것은 과한 구현.                                               |

### 결과
서버 재시작 시 큐에 남아 있던 이벤트가 유실될 수 있다.
